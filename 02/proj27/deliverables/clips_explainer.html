A line-by-line walkthrough of both finite state machine implementations in CLIPS.

1
Templates — Defining the Data Structures
▼
1
;;; Templates
2
(deftemplate state (slot name))
3
(deftemplate arc (slot from) (slot to) (slot input))
What this does
deftemplate works like a struct in C — it defines the shape of a fact.

• (deftemplate state (slot name)) — creates a fact type with one field: name. At any moment, exactly one state fact exists in memory, holding the machine's current state (e.g. start, s5, success).

• (deftemplate arc (slot from) (slot to) (slot input)) — creates a fact type with three fields describing a transition: where you come from, where you go, and what input triggers it.
2
fsm-transition — The Core Engine Rule
▼
5
(defrule fsm-transition
6
?current <- (state (name ?s))
7
?inp <- (input ?value)
8
(arc (from ?s) (to ?next) (input ?value))
9
=>
10
(retract ?current) ; remove old state
11
(retract ?inp) ; consume the input
12
(assert (state (name ?next)))
13
(printout t "Transition: " ?s " --[" ?value "]--> " ?next crlf))
Line-by-line breakdown
Line 6 — Match current state: ?current <- binds the actual fact object to ?current so we can retract it later. ?s captures its name (e.g. start).

Line 7 — Match input fact: Looks for an (input ?value) fact in memory. ?value will be N or Q.

Line 8 — Find matching arc: Looks for an arc fact whose from equals ?s AND whose input equals ?value. If found, ?next is bound to the destination state. This is pattern matching — CLIPS only fires the rule when ALL three conditions are true simultaneously.

Line 10 — Remove old state: Retracts the current state fact — the machine can only be in one state at a time.

Line 11 — Consume input: Retracts the input fact so the rule doesn't re-fire on the same input infinitely.

Line 12 — Move to next state: Asserts a new state fact with the destination name.

Line 13 — Print transition: Logs the transition for tracing, e.g. Transition: start --[N]--> s5.
state: start
+
(input N)
+
(arc start→s5 N)
→
state: s5
3
fsm-no-transition — Error Handling
▼
16
(defrule fsm-no-transition
17
(state (name ?s))
18
(input ?value)
19
(not (arc (from ?s) (to ?) (input ?value)))
20
=>
21
(printout t "ERROR: No transition from " ?s " on input '" ?value "'" crlf))
What this does
This rule is the mirror image of the transition rule — it fires when there is a state and an input, but no matching arc exists.

The key is (not (arc ...)) — CLIPS's logical negation. The ? wildcard in (to ?) means "any value" — we only care that no arc from ?s on input ?value exists at all.

Since the vending machine only accepts N (nickel) and Q (quarter), asserting (input D) (a dime) would trigger this rule.
⚠️
Note: Both rules cannot fire at the same time. If a valid arc exists, fsm-transition fires. If no arc exists, fsm-no-transition fires. They are mutually exclusive by design.
4
deffacts — Vending Machine Arc Table
▼
How it works
deffacts preloads facts into working memory on (reset). Each arc fact is one row of the FSM's transition table. The rules above read these facts at runtime — the rules never change, only the arc facts do.
From StateInputTo StateCredit Change
startN→s50¢ + 5¢
startQ→s100¢ + 10¢
s5N→s105¢ + 5¢
s5Q→s155¢ + 10¢
s10N→s1510¢ + 5¢
s10Q→s2010¢ + 10¢
s15N→s2015¢ + 5¢
s15Q→success15¢ + 10¢ = 25¢ ✓
s20N→success20¢ + 5¢ = 25¢ ✓
s20Q→success20¢ + 10¢ ≥ 25¢ ✓
💡
A quarter is modelled as +10¢ (not 25¢) because the machine only needs 5¢ increments to reach exactly 25¢. Each state name directly encodes the accumulated credit: s5 = 5¢, s10 = 10¢, etc.
1
Templates & Rules — Reused from Vending Machine
▼
Key insight
The deftemplate definitions and both rules (fsm-transition, fsm-no-transition) are identical to the vending machine file. This is the power of the generic FSM design:

The rules describe behaviour (how to move between states).
The arc facts describe the machine (which transitions are valid).

Swapping the arc facts turns the same engine into a completely different machine.
🔁
In a real application you could put both machines' arcs in one file with a (machine ?m) slot on each arc, and filter by machine name — one engine, infinite FSMs.

State naming convention
Each state name is a prefix log of characters seen so far. sBEG means the machine has accepted B, E, G in order and is waiting for I.

This makes the arc table self-documenting — you can read the path just from the state names without needing to draw the diagram.

WHILE path — W → H → I → L → E
▼

(arc (from sWHIL) (to success) (input E))
Shared entry point with WRITE
Both WHILE and WRITE start with W, so they both transition from start into the shared state sW. The machine doesn't yet know which word is coming — it just knows a W was seen.

The paths diverge at the second character: H leads toward WHILE, R leads toward WRITE. From sW, asserting (input H) goes to sWH and asserting (input R) goes to sWR.

Divergence from sW
From the shared state sW, an R input leads down the WRITE branch. From this point the states are fully independent from the WHILE path.

Notice that I appears in both WHILE (sWH → sWHI) and WRITE (sWR → sWRI) — but they are different states so there's no ambiguity. The machine tracks the full history of characters in the state name.
